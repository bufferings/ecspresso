4.2 複数環境での運用

多くの組織では、開発、テスト、ステージング、本番など、複数の環境でアプリケーションを運用しています。ecspressoを使用して、これらの環境間で一貫したデプロイを実現する方法を説明します。

環境別の設定ファイル管理

flowchart TD Base[共通設定\necspresso.yml] --> Dev[開発環境\necspresso.dev.yml] Base --> Staging[ステージング環境\necspresso.staging.yml] Base --> Prod[本番環境\necspresso.prod.yml]

1. 環境別の設定ファイル

各環境ごとに別々の設定ファイルを作成します。

# ecspresso.dev.yml
region: ap-northeast-1
cluster: app-dev
service: web-service-dev
service_definition: ecs-service-def.json
task_definition: ecs-task-def.json
timeout: 5m

# ecspresso.staging.yml
region: ap-northeast-1
cluster: app-staging
service: web-service-staging
service_definition: ecs-service-def.json
task_definition: ecs-task-def.json
timeout: 10m

# ecspresso.prod.yml
region: ap-northeast-1
cluster: app-prod
service: web-service-prod
service_definition: ecs-service-def.json
task_definition: ecs-task-def.json
timeout: 15m

2. 環境変数による設定ファイルの選択

環境変数を使用して、使用する設定ファイルを選択します。

$ ENV=dev ecspresso --config=ecspresso.$ENV.yml deploy

タスク定義とサービス定義のテンプレート化

タスク定義とサービス定義をテンプレート化して、環境ごとの違いを管理します。

1. 環境変数の使用

タスク定義内で環境変数を使用します。

{
  "family": "web-service-{{env}}",
  "containerDefinitions": [
    {
      "name": "web",
      "image": "{{ecr_repository}}:{{image_tag}}",
      "environment": [
        {
          "name": "APP_ENV",
          "value": "{{env}}"
        },
        {
          "name": "LOG_LEVEL",
          "value": "{{log_level}}"
        }
      ],
      "logConfiguration": {
        "logDriver": "awslogs",
        "options": {
          "awslogs-group": "/ecs/web-service-{{env}}",
          "awslogs-region": "{{region}}",
          "awslogs-stream-prefix": "ecs"
        }
      }
    }
  ]
}

2. 環境別の変数ファイル

環境ごとに変数ファイルを作成します。

# dev.env
env=dev
ecr_repository=123456789012.dkr.ecr.ap-northeast-1.amazonaws.com/web-service
image_tag=latest
log_level=debug
region=ap-northeast-1

# staging.env
env=staging
ecr_repository=123456789012.dkr.ecr.ap-northeast-1.amazonaws.com/web-service
image_tag=stable
log_level=info
region=ap-northeast-1

# prod.env
env=prod
ecr_repository=123456789012.dkr.ecr.ap-northeast-1.amazonaws.com/web-service
image_tag=v1.0.0
log_level=warn
region=ap-northeast-1

3. 環境ファイルの使用

ecspressoコマンドで環境ファイルを指定します。

$ ecspresso --config=ecspresso.dev.yml --env-file=dev.env deploy

プラグインを使用した設定管理

ecspressoのプラグイン機能を使用して、Terraform状態ファイルやCloudFormationスタックから値を取得できます。

1. Terraformプラグインの設定

# ecspresso.yml
region: ap-northeast-1
cluster: app-{{env}}
service: web-service-{{env}}
service_definition: ecs-service-def.json
task_definition: ecs-task-def.json
timeout: 10m
plugins:
  - name: tfstate
    config:
      path: terraform.tfstate
      vars:
        vpc_id: $.outputs.vpc_id.value
        subnet_ids: $.outputs.private_subnet_ids.value

2. CloudFormationプラグインの設定

# ecspresso.yml
region: ap-northeast-1
cluster: app-{{env}}
service: web-service-{{env}}
service_definition: ecs-service-def.json
task_definition: ecs-task-def.json
timeout: 10m
plugins:
  - name: cloudformation
    config:
      stack_name: network-{{env}}
      vars:
        vpc_id: VpcId
        subnet_ids: PrivateSubnetIds

環境間のプロモーション戦略

アプリケーションを開発環境からステージング、本番環境へと段階的にプロモーションする戦略を説明します。

flowchart LR Dev[開発環境] -->|テスト合格| Staging[ステージング環境] Staging -->|受入テスト合格| Prod[本番環境]

1. イメージタグ戦略

環境間でイメージを昇格させる方法：

  • 開発環境: 最新のコミットからビルドされたイメージ（latestタグ）
  • ステージング環境: 開発環境でテスト済みのイメージ（stableタグ）
  • 本番環境: ステージング環境で検証済みのイメージ（v1.0.0などのバージョンタグ）

2. イメージのプロモーションスクリプト例

#!/bin/bash
# 開発環境からステージング環境へのプロモーション
SOURCE_TAG="latest"
TARGET_TAG="stable"
ECR_REPO="123456789012.dkr.ecr.ap-northeast-1.amazonaws.com/web-service"

# イメージのプル
docker pull $ECR_REPO:$SOURCE_TAG

# 新しいタグを付ける
docker tag $ECR_REPO:$SOURCE_TAG $ECR_REPO:$TARGET_TAG

# イメージのプッシュ
docker push $ECR_REPO:$TARGET_TAG

# ステージング環境にデプロイ
ecspresso --config=ecspresso.staging.yml --env-file=staging.env deploy

環境固有の考慮事項

1. スケーリング設定

環境ごとに異なるスケーリング設定を適用します。

# 開発環境（最小限のリソース）
$ ecspresso --config=ecspresso.dev.yml deploy --tasks=1

# ステージング環境（中程度のリソース）
$ ecspresso --config=ecspresso.staging.yml deploy --tasks=2

# 本番環境（高可用性のためのリソース）
$ ecspresso --config=ecspresso.prod.yml deploy --tasks=3

2. デプロイ戦略

環境ごとに異なるデプロイ戦略を適用します。

  • 開発環境: ECSローリングアップデート（迅速なデプロイ）
  • ステージング環境: ECSローリングアップデート（検証用）
  • 本番環境: CodeDeployによるBlue/Greenデプロイ（安全なデプロイ）

3. モニタリングとアラート

環境ごとに異なるモニタリング設定を適用します。

  • 開発環境: 最小限のアラート（重大なエラーのみ）
  • ステージング環境: 中程度のアラート（エラーと警告）
  • 本番環境: 包括的なアラート（エラー、警告、パフォーマンス指標）
