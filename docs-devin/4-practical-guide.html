4.1 CI/CDパイプラインとの統合

目次

概要

ecspressoをCI/CDパイプラインに統合することで、ECSサービスのデプロイメントを自動化し、開発ワークフローを効率化できます。主な統合ステップは以下の通りです：

  1. CI/CD環境にecspressoをインストールする
  2. AWS認証情報を設定する
  3. リソースを検証する（verify）
  4. 変更を確認する（diff）
  5. デプロイを実行する（deploy）
  6. デプロイ結果を監視する
sequenceDiagram actor Dev as 開発者 participant Git as Gitリポジトリ participant CI as CI/CDパイプライン participant Ecspresso as ecspresso participant AWS as AWS ECS Dev->>Git: コード変更をプッシュ Git->>CI: CI/CDパイプラインをトリガー CI->>CI: AWS認証情報を設定 CI->>Ecspresso: ecspresso verify Ecspresso->>AWS: リソース検証 AWS-->>Ecspresso: 検証結果 CI->>Ecspresso: ecspresso diff Ecspresso->>AWS: 設定差分確認 AWS-->>Ecspresso: 差分結果 CI->>Ecspresso: ecspresso deploy Ecspresso->>AWS: デプロイ実行 AWS-->>Ecspresso: デプロイ結果 CI->>AWS: デプロイ結果を監視 AWS-->>CI: サービス状態 CI-->>Dev: デプロイ結果通知

GitHub Actionsでの統合

GitHub Actionsを使用してecspressoを統合する例：

name: Deploy to ECS

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v1
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: ap-northeast-1
      
      - name: Install ecspresso
        uses: kayac/ecspresso@v2
        with:
          version: v2.5.0
      
      - name: Verify resources
        run: ecspresso verify --config ecspresso.yml
      
      - name: Show diff
        run: ecspresso diff --config ecspresso.yml
      
      - name: Deploy to ECS
        if: github.ref == 'refs/heads/main' && github.event_name == 'push'
        run: ecspresso deploy --config ecspresso.yml

GitLab CIでの統合

GitLab CIを使用してecspressoを統合する例：

stages:
  - verify
  - deploy

variables:
  ECSPRESSO_VERSION: v2.5.0

before_script:
  - curl -sL https://github.com/kayac/ecspresso/releases/download/${ECSPRESSO_VERSION}/ecspresso_${ECSPRESSO_VERSION}_linux_amd64.tar.gz | tar xz
  - mv ecspresso /usr/local/bin/

verify:
  stage: verify
  script:
    - ecspresso verify --config ecspresso.yml
    - ecspresso diff --config ecspresso.yml
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'

deploy:
  stage: deploy
  script:
    - ecspresso deploy --config ecspresso.yml
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'

CircleCIでの統合

CircleCIを使用してecspressoを統合する例：

version: 2.1

orbs:
  aws-cli: circleci/aws-cli@2.0.3
  ecspresso: fujiwara/ecspresso@2.0.4

jobs:
  verify:
    docker:
      - image: cimg/base:2022.06
    steps:
      - checkout
      - aws-cli/setup:
          profile-name: default
      - ecspresso/install:
          version: v2.5.0
      - run:
          name: Verify resources
          command: ecspresso verify --config ecspresso.yml
      - run:
          name: Show diff
          command: ecspresso diff --config ecspresso.yml

  deploy:
    docker:
      - image: cimg/base:2022.06
    steps:
      - checkout
      - aws-cli/setup:
          profile-name: default
      - ecspresso/install:
          version: v2.5.0
      - run:
          name: Deploy to ECS
          command: ecspresso deploy --config ecspresso.yml

workflows:
  version: 2
  deploy-workflow:
    jobs:
      - verify
      - deploy:
          requires:
            - verify
          filters:
            branches:
              only: main

ベストプラクティス

  • 環境変数の活用: 環境ごとに異なる設定を環境変数で管理し、--envfileオプションで読み込む
  • 検証の自動化: PRごとにverifyとdiffを実行して問題を早期に発見する
  • デプロイの自動化: メインブランチへのマージ時に自動的にデプロイする
  • ロールバックの準備: 問題が発生した場合に備えて、ロールバックの手順を用意する
  • 通知の設定: デプロイの成功・失敗をSlackなどに通知する
  • バージョン固定: ecspressoのバージョンを固定して、予期しない動作を防ぐ
  • セキュリティ対策: AWS認証情報をシークレットとして安全に管理する

複数環境の課題

開発、ステージング、本番など複数の環境でecspressoを使用する際の主な課題：

  • 環境ごとに異なる設定の管理
  • 環境間での一貫性の確保
  • 環境固有の値（ARN、シークレットなど）の管理
  • デプロイプロセスの標準化

環境管理の方法

ecspressoで複数環境を管理するための主な方法：

1. 環境ごとの設定ファイル

環境ごとに別々の設定ファイルを用意する方法：

project/
  ├── dev/
  │   ├── ecspresso.yml
  │   ├── ecs-service-def.json
  │   └── ecs-task-def.json
  ├── staging/
  │   ├── ecspresso.yml
  │   ├── ecs-service-def.json
  │   └── ecs-task-def.json
  └── prod/
      ├── ecspresso.yml
      ├── ecs-service-def.json
      └── ecs-task-def.json

2. 環境変数ファイル

共通の設定ファイルと環境ごとの環境変数ファイルを組み合わせる方法：

project/
  ├── ecspresso.yml
  ├── ecs-service-def.json
  ├── ecs-task-def.json
  ├── dev.env
  ├── staging.env
  └── prod.env

環境変数ファイルの例（dev.env）：

CLUSTER_NAME=dev-cluster
SERVICE_NAME=dev-service
DESIRED_COUNT=1
CONTAINER_IMAGE=myapp:dev
LOG_GROUP=/ecs/dev/myapp

コマンド実行例：

$ ecspresso --envfile dev.env deploy

3. Jsonnetによる環境分離

Jsonnetを使用して環境ごとの設定を生成する方法：

project/
  ├── ecspresso.jsonnet
  ├── ecs-service-def.jsonnet
  ├── ecs-task-def.jsonnet
  └── environments/
      ├── dev.libsonnet
      ├── staging.libsonnet
      └── prod.libsonnet

Jsonnetによる環境分離

Jsonnetを使用して環境ごとの設定を生成する例：

// environments/dev.libsonnet
{
  env: 'dev',
  cluster: 'dev-cluster',
  service: 'myapp-dev',
  desired_count: 1,
  container_image: 'myapp:dev',
  log_group: '/ecs/dev/myapp',
  memory: 512,
  cpu: 256,
}

// ecs-task-def.jsonnet
local env = import 'environments/' + std.extVar('env') + '.libsonnet';

{
  family: env.service,
  executionRoleArn: 'arn:aws:iam::123456789012:role/ecsTaskExecutionRole',
  taskRoleArn: 'arn:aws:iam::123456789012:role/' + env.service + '-task-role',
  containerDefinitions: [
    {
      name: 'app',
      image: env.container_image,
      essential: true,
      logConfiguration: {
        logDriver: 'awslogs',
        options: {
          'awslogs-group': env.log_group,
          'awslogs-region': 'ap-northeast-1',
          'awslogs-stream-prefix': 'ecs'
        }
      },
      environment: [
        {
          name: 'ENV',
          value: env.env
        }
      ],
      memory: env.memory,
      cpu: env.cpu
    }
  ]
}

コマンド実行例：

$ ecspresso --config ecspresso.jsonnet --ext-str env=dev deploy

ベストプラクティス

  • 環境設定の分離: 環境固有の設定を明確に分離する
  • 共通設定の再利用: 環境間で共通の設定を再利用する
  • シークレット管理: AWS Secrets ManagerやSSM Parameter Storeを使用してシークレットを管理する
  • プラグインの活用: CloudFormationやTerraformのプラグインを使用してリソース情報を取得する
  • デプロイスクリプト: 環境ごとのデプロイを自動化するスクリプトを作成する
  • 環境変数の検証: 必須の環境変数が設定されているか検証する

複数マイクロサービスのデプロイスクリプト例：

#!/bin/bash
set -e

ENV=$1
SERVICES=("service-a" "service-b" "service-c")

if [ -z "$ENV" ]; then
  echo "Usage: $0 "
  exit 1
fi

for SERVICE in "${SERVICES[@]}"; do
  echo "Deploying $SERVICE to $ENV environment..."
  cd "$SERVICE"
  ecspresso --envfile "../environments/$ENV.env" verify
  ecspresso --envfile "../environments/$ENV.env" diff
  ecspresso --envfile "../environments/$ENV.env" deploy
  cd ..
done

echo "All services deployed successfully to $ENV environment."

ベストプラクティス

  • サービス構成の標準化: 一貫したサービス構成テンプレートを使用する
  • デプロイの自動化: デプロイプロセスを完全に自動化する
  • 段階的デプロイ: 重要度の低いサービスから順にデプロイする

  • ロールバック計画: 問題発生時のロールバック手順を用意する
  • ドキュメント化: サービス構成とデプロイプロセスをドキュメント化する

複数マイクロサービスのデプロイスクリプト例：

#!/bin/bash
set -e

ENV=$1
SERVICE=$2
ACTION=${3:-deploy}
SERVICES=("api" "worker" "frontend")

if [ -z "$ENV" ]; then
  echo "Usage: $0  [service] [action]"
  echo "  env: dev, staging, prod"
  echo "  service: all, api, worker, frontend"
  echo "  action: deploy, rollback, scale, status"
  exit 1
fi

if [ "$SERVICE" = "all" ] || [ -z "$SERVICE" ]; then
  DEPLOY_SERVICES=("${SERVICES[@]}")
else
  DEPLOY_SERVICES=("$SERVICE")
fi

for SVC in "${DEPLOY_SERVICES[@]}"; do
  echo "Performing $ACTION on $SVC in $ENV environment..."
  cd "services/$SVC"
  
  case "$ACTION" in
    deploy)
      ecspresso --envfile "../../environments/$ENV.env" verify
      ecspresso --envfile "../../environments/$ENV.env" diff
      ecspresso --envfile "../../environments/$ENV.env" deploy
      ;;
    rollback)
      ecspresso --envfile "../../environments/$ENV.env" rollback
      ;;
    scale)
      read -p "Enter desired task count for $SVC: " TASKS
      ecspresso --envfile "../../environments/$ENV.env" scale --tasks $TASKS
      ;;
    status)
      ecspresso --envfile "../../environments/$ENV.env" status --events 10 --tasks
      ;;
    *)
      echo "Unknown action: $ACTION"
      exit 1
      ;;
  esac
  
  cd ../..
done

echo "All operations completed successfully."
