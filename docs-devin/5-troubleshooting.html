認証エラー

AWS認証情報が正しく設定されていない場合に発生するエラー：

エラー例：

Error: NoCredentialProviders: no valid providers in chain. Deprecated.
For verbose messaging see aws.Config.CredentialsChainVerboseErrors

解決策：

  1. AWS認証情報が正しく設定されているか確認します： 
    $ aws configure list
  2. 環境変数を設定します： 
    $ export AWS_ACCESS_KEY_ID=your_access_key
$ export AWS_SECRET_ACCESS_KEY=your_secret_key
$ export AWS_REGION=ap-northeast-1
  3. または、共有認証情報ファイルを設定します： 
    $ aws configure
  4. プロファイルを指定してecspressoを実行します： 
    $ ecspresso --profile your_profile deploy

権限エラー

必要なAWS権限がない場合に発生するエラー：

エラー例：

Error: AccessDeniedException: User: arn:aws:iam::123456789012:user/username is not authorized to perform: ecs:DescribeServices on resource: arn:aws:ecs:ap-northeast-1:123456789012:service/cluster-name/service-name

解決策：

  1. 必要な権限を持つIAMポリシーを確認します： 
    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "ecs:DescribeServices",
        "ecs:DescribeTaskDefinition",
        "ecs:RegisterTaskDefinition",
        "ecs:UpdateService",
        "ecs:RunTask",
        "ecs:DescribeTasks",
        "ecs:StopTask",
        "ecs:ListTasks",
        "iam:PassRole"
      ],
      "Resource": "*"
    }
  ]
}
  2. IAMユーザーまたはロールに必要なポリシーをアタッチします。
  3. 特定のリソースに対する権限を制限する場合は、Resourceセクションを適切に設定します。

リージョンエラー

AWSリージョンが正しく設定されていない場合に発生するエラー：

エラー例：

Error: ClusterNotFoundException: Cluster not found.

解決策：

  1. 設定ファイルでリージョンを指定します： 
    # ecspresso.yml
region: ap-northeast-1
cluster: your-cluster
service: your-service
  2. コマンドラインでリージョンを指定します： 
    $ ecspresso --region ap-northeast-1 deploy
  3. 環境変数でリージョンを設定します： 
    $ export AWS_REGION=ap-northeast-1

サービスエラー

サービス定義に問題がある場合に発生するエラー：

エラー例：

Error: InvalidParameterException: The target group with targetGroupArn arn:aws:elasticloadbalancing:ap-northeast-1:123456789012:targetgroup/tg-name/abcdef1234567890 does not exist.

解決策：

  1. サービス定義のリソース参照（ターゲットグループ、セキュリティグループなど）が正しいか確認します。
  2. verifyコマンドを使用してリソースを検証します： 
    $ ecspresso verify
  3. diffコマンドを使用して変更内容を確認します： 
    $ ecspresso diff

タスク定義エラー

タスク定義に問題がある場合に発生するエラー：

エラー例：

Error: ClientException: Invalid setting for container 'web'. At least one of 'essential' or 'dependsOn' or 'links' must be specified.

解決策：

  1. タスク定義の必須フィールドが正しく設定されているか確認します。
  2. renderコマンドを使用してタスク定義をレンダリングして確認します： 
    $ ecspresso render --task-definition
  3. AWS ECSのドキュメントを参照して、タスク定義の要件を確認します。

デプロイタイムアウト

デプロイが長時間完了しない場合に発生するエラー：

エラー例：

Error: context deadline exceeded

解決策：

  1. タイムアウト時間を延長します： 
    $ ecspresso deploy --timeout 1200
  2. サービスのイベントを確認して問題を特定します： 
    $ ecspresso status --events 20
  3. デプロイの待機をスキップして、後で状態を確認します： 
    $ ecspresso deploy --no-wait-until-stable
$ ecspresso status

ロールバックの問題

ロールバック時に発生する可能性のある問題：

エラー例：

Error: TaskDefinitionNotFoundException: The specified task definition does not exist.

解決策：

  1. 以前のタスク定義が存在するか確認します： 
    $ aws ecs list-task-definitions --family-prefix your-family
  2. 特定のリビジョンを指定してロールバックします： 
    $ ecspresso rollback --revision 10
  3. タスク定義の登録解除を避けるために、deregisterオプションを使用しないようにします： 
    $ ecspresso rollback --deregister=false

タスク起動エラー

タスクの起動に失敗する場合に発生するエラー：

エラー例：

Error: InvalidParameterException: The specified subnet does not exist.

解決策：

  1. ネットワーク設定が正しいか確認します： 
    $ ecspresso verify
  2. ネットワーク設定を指定してタスクを実行します： 
    $ ecspresso run --network-configuration network.json
  3. network.jsonの例： 
    {
  "awsvpcConfiguration": {
    "subnets": [
      "subnet-12345678",
      "subnet-87654321"
    ],
    "securityGroups": [
      "sg-12345678"
    ],
    "assignPublicIp": "ENABLED"
  }
}

execコマンドエラー

execコマンドの実行に失敗する場合に発生するエラー：

エラー例：

Error: ExecuteCommandAgentError: The execute command failed due to an error from the SSM agent on the container: SessionManagerPlugin is not found. Please refer to SessionManager Documentation here: http://docs.aws.amazon.com/console/systems-manager/session-manager-plugin-not-found

解決策：

  1. Session Manager Pluginがインストールされているか確認します： 
    $ session-manager-plugin --version
  2. Session Manager Pluginをインストールします： 
    # macOS
$ brew install session-manager-plugin

# Linux
$ curl "https://s3.amazonaws.com/session-manager-downloads/plugin/latest/linux_64bit/session-manager-plugin.rpm" -o "session-manager-plugin.rpm"
$ sudo yum install -y session-manager-plugin.rpm
  3. タスク定義でECS Execが有効になっているか確認します： 
    # ecspresso.yml
service_definition: ecs-service-def.json

# ecs-service-def.json
{
  "enableExecuteCommand": true
}

コンテナエラー

コンテナの起動や実行に問題がある場合：

エラー例：

Task failed to start: ResourceInitializationError: unable to pull secrets or registry auth: execution resource retrieval failed: unable to retrieve ecr registry auth: service call has been retried 3 time(s): RequestError: send request failed caused by: Post "https://api.ecr.ap-northeast-1.amazonaws.com/": dial tcp: lookup api.ecr.ap-northeast-1.amazonaws.com: no such host

解決策：

  1. タスクの状態を確認します： 
    $ ecspresso tasks
  2. タスクのログを確認します： 
    $ ecspresso run --wait --watch-container web
  3. コンテナ内でコマンドを実行して問題を診断します： 
    $ ecspresso exec --interactive -- /bin/bash
  4. ネットワーク接続の問題がある場合は、VPCエンドポイントの設定を確認します。

YAML構文エラー

設定ファイルのYAML構文に問題がある場合：

エラー例：

Error: yaml: line 5: did not find expected key

解決策：

  1. YAMLの構文を確認します。特にインデントに注意してください。
  2. YAMLバリデーターを使用して構文を検証します： 
    $ yamllint ecspresso.yml
  3. 正しいYAML形式の例： 
    region: ap-northeast-1
cluster: my-cluster
service: my-service
service_definition: ecs-service-def.json
task_definition: ecs-task-def.json
timeout: 600

テンプレート関数エラー

テンプレート関数の使用に問題がある場合：

エラー例：

Error: template: :1:3: executing "" at <env "MISSING_ENV">: map has no entry for key "MISSING_ENV"

解決策：

  1. 必要な環境変数が設定されているか確認します： 
    $ export MISSING_ENV=value
  2. デフォルト値を使用してエラーを回避します： 
    {{ env `MISSING_ENV` | default "default_value" }}
  3. renderコマンドを使用して、テンプレートのレンダリング結果を確認します： 
    $ ecspresso render --task-definition

プラグインエラー

プラグインの使用に問題がある場合：

エラー例：

Error: failed to setup tfstate plugin: failed to load tfstate: open tfstate.json: no such file or directory

解決策：

  1. プラグイン設定が正しいか確認します： 
    # ecspresso.yml
plugins:
  tfstate:
    path: tfstate.json
  2. 必要なファイルが存在するか確認します： 
    $ ls -la tfstate.json
  3. プラグインの設定を修正します： 
    # ecspresso.yml
plugins:
  tfstate:
    path: /path/to/tfstate.json

Jsonnetエラー

Jsonnetファイルの使用に問題がある場合：

エラー例：

Error: RUNTIME ERROR: couldn't open import "environments/dev.libsonnet": no match locally or in the Jsonnet library paths

解決策：

  1. インポートするファイルが存在するか確認します： 
    $ ls -la environments/dev.libsonnet
  2. Jsonnetのライブラリパスを設定します： 
    $ export JSONNET_PATH=./lib
  3. Jsonnetファイルの構文を確認します： 
    $ jsonnetfmt --test ecs-task-def.jsonnet

デバッグモード

ecspressoのデバッグモードを有効にして詳細な情報を取得する方法：

$ ecspresso --debug deploy

デバッグモードでは、以下の情報が表示されます：

  • AWS APIリクエストとレスポンス
  • テンプレートのレンダリング結果
  • 内部処理の詳細

ログ分析

ECSタスクのログを分析する方法：

  1. タスクのログを表示します： 
    $ ecspresso run --wait --watch-container web
  2. AWS Management Consoleでログを確認します：
    1. ECSコンソールでタスクを選択
    2. 「ログ」タブを選択
    3. CloudWatch Logsでログを確認
  3. AWS CLIでログを取得します： 
    $ aws logs get-log-events --log-group-name /ecs/my-service --log-stream-name ecs/container-name/task-id

AWS CLIを使用したデバッグ

AWS CLIを使用してECSリソースを確認する方法：

  1. サービスの状態を確認します： 
    $ aws ecs describe-services --cluster my-cluster --services my-service
  2. タスク定義を確認します： 
    $ aws ecs describe-task-definition --task-definition my-task-family:1
  3. 実行中のタスクを確認します： 
    $ aws ecs list-tasks --cluster my-cluster --service-name my-service
  4. タスクの詳細を確認します： 
    $ aws ecs describe-tasks --cluster my-cluster --tasks task-id

デプロイ前の検証

デプロイ前に問題を検出するための検証手順：

  1. リソースを検証します： 
    $ ecspresso verify
  2. 変更内容を確認します： 
    $ ecspresso diff
  3. タスク定義をレンダリングして確認します： 
    $ ecspresso render --task-definition
  4. サービス定義をレンダリングして確認します： 
    $ ecspresso render --service-definition
  5. ドライランモードでデプロイをシミュレーションします： 
    $ ecspresso deploy --dry-run

よくある問題の解決策

ecspressoを使用する際によく発生する問題とその解決策：

問題 解決策
AWS認証エラー AWS認証情報を正しく設定する（環境変数、共有認証情報ファイル、IAMロール）
リソースが見つからない verifyコマンドを使用してリソースの存在を確認する
デプロイタイムアウト タイムアウト時間を延長するか、--no-wait-until-stableオプションを使用する
テンプレート関数エラー 環境変数を設定するか、デフォルト値を使用する
コンテナ起動エラー タスクのログを確認し、コンテナの設定を修正する
ネットワーク設定エラー サブネットとセキュリティグループの設定を確認する
プラグインエラー プラグインの設定と必要なファイルの存在を確認する

ベストプラクティス

ecspressoを効果的に使用するためのベストプラクティス：

  • バージョン管理: 設定ファイルをバージョン管理システム（Git）で管理する
  • 環境変数の活用: 環境固有の値を環境変数で管理する
  • デプロイ前の検証: verifyとdiffコマンドを使用して問題を早期に検出する
  • ドライラン: 重要な変更を行う前にドライランモードでシミュレーションする
  • ロールバック計画: 問題発生時のロールバック手順を用意する
  • CI/CD統合: 自動化されたパイプラインでデプロイを行う
  • モニタリング: CloudWatchアラームとダッシュボードを設定する
  • ドキュメント化: デプロイプロセスとトラブルシューティング手順をドキュメント化する

これらのベストプラクティスを実践することで、ecspressoを使用したECSリソースの管理がより効率的かつ安全になります。
