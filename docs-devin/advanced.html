バージョン制約

設定ファイル内のrequired_versionは、ecspressoのバージョンを固定するためのものです。この機能はconfig.goのConfig構造体で実装されています。

required_version: ">= 2.0.0, < 3"

これにより、バージョンが2.0.0以上かつ3未満の場合にのみecspressoが実行できるようになります。バージョンがこの範囲内にない場合、実行は失敗します。

内部的にはgo-versionライブラリを使用して、バージョン制約を解析し、現在のバージョンが制約を満たしているかどうかを確認します。App.CheckVersionメソッドが実際の検証を行います。

flowchart LR Config[設定ファイル] --> |required_version| Check[バージョン検証] Check --> |一致| Execute[実行継続] Check --> |不一致| Error[エラー終了]

Application Auto Scalingの管理

Application Auto Scalingを使用するECSサービスの場合、ecspresso scaleコマンドを使用して自動スケーリングの最小値と最大値の設定を調整することが簡単にできます。この機能はdeploy.goのApp.DeployメソッドとAutoScalingTargetResource構造体で実装されています。

$ ecspresso scale --tasks 5 --auto-scaling-min 5 --auto-scaling-max 20

内部的には、以下のステップで処理されます：

  1. サービスのスケーラブルターゲットを特定（findScalableTargetメソッド）
  2. 最小・最大キャパシティの設定（RegisterScalableTarget API）
  3. 必要に応じて一時停止状態の更新
sequenceDiagram participant User as ユーザー participant App as ecspresso participant AWS as AWS API User->>App: ecspresso scale App->>AWS: DescribeScalableTargets AWS-->>App: スケーラブルターゲット情報 App->>AWS: RegisterScalableTarget AWS-->>App: 登録結果 opt 一時停止/再開が指定された場合 App->>AWS: UpdateScalableTargetSuspendedState AWS-->>App: 更新結果 end App-->>User: 完了メッセージ

ecspresso deployとscaleはアプリケーションの自動スケーリングを一時停止および再開できます：

  • --suspend-auto-scalingは一時停止状態をtrueに設定します（SuspendedState構造体）
  • --resume-auto-scalingは一時停止状態をfalseに設定します（SuspendedState構造体）

JSONとYAMLの代わりにJsonnetを使用する

ecspressoはJsonnetファイル形式をサポートしています。この機能はconfig.goのnewLoader関数とjsonnetLoader構造体で実装されています。

  • v1.7以降：サービスとタスク定義のJsonnetサポート
  • v2.0以降：設定ファイルのJsonnetサポート
  • v2.4以降：Jsonnet関数をサポート（jsonnet.goで実装）
flowchart TD File[ファイル] --> Extension{拡張子は.jsonnet?} Extension -->|Yes| Jsonnet[Jsonnetとして処理] Extension -->|No| Normal[通常のJSONまたはYAML] Jsonnet --> Convert[JSONに変換] Convert --> Template[テンプレート構文評価] Normal --> Template Template --> Result[最終結果]

ファイルの拡張子が.jsonnetの場合、ecspressoは以下の順序で処理します：

  1. Jsonnetとして処理（github.com/google/go-jsonnetライブラリを使用）
  2. JSONに変換（jsonnetLoader.Loadメソッド）
  3. テンプレート構文の評価でロード（github.com/kayac/go-configライブラリを使用）

Jsonnet関数

v2.4以降はJsonnetファイル内でJsonnetネイティブ関数をサポートしています。この機能はjsonnet.goのregisterNativeFuncs関数で実装されています。

.jsonnetファイル内で：

  1. local func = std.native('func');を定義
  2. func()を使用

Jsonnet関数はJsonnetファイルのレンダリング時に評価され、テンプレート構文との競合を回避するのに役立ちます。内部的には、jsonnet.VMオブジェクトに関数が登録され、Jsonnetの評価時に呼び出されます。

env、must_env

local env = std.native('env');
local must_env = std.native('must_env');
{
  foo: env('FOO', 'default value'),
  bar: must_env('BAR'),
  bazNumber: std.parseInt(env('BAZ_NUMBER', '0')),
  booBool: std.parseJson(env('BOO_BOOL', 'false')),
}

これらの関数はjsonnet.goのenvFuncとmustEnvFunc関数として実装されており、環境変数の値を取得するために使用されます。must_envは環境変数が存在しない場合にエラーを発生させます。

Fargateへのデプロイ

Fargateにサービスをデプロイする場合、タスク定義とサービス定義の両方に特定の設定が必要です。この機能はdeploy.goのApp.Deployメソッドとrun.goのApp.Runメソッドで実装されています。

classDiagram class TaskDefinition { +requiresCompatibilities: ["FARGATE"] +networkMode: "awsvpc" +cpu: string +memory: string +executionRoleArn: string } class Service { +launchType: "FARGATE" +networkConfiguration: object } TaskDefinition --> Service : デプロイ時に使用

タスク定義の場合：

  • requiresCompatibilities（「FARGATE」が必要）- TaskDefinition構造体で検証
  • networkMode（「awsvpc」が必要）- TaskDefinition構造体で検証
  • cpu（必須）- TaskDefinition構造体で検証
  • memory（必須）- TaskDefinition構造体で検証
  • executionRoleArn（オプション）- TaskDefinition構造体で検証

サービス定義の場合：

  • launchType（「FARGATE」が必要）- Service構造体で検証
  • networkConfiguration（「awsvpcConfiguration」が必要）- Service構造体で検証

内部的には、verify.goのverifyTaskDefinitionメソッドとverifyServiceメソッドが、Fargateの要件を満たしているかどうかを検証します。要件が満たされていない場合、デプロイ前に検証エラーが発生します。

ECS Service Connectのサポート

ecspressoはECS Service Connectをサポートしています。この機能はdeploy.goのApp.DeployメソッドとService構造体で実装されています。

設定するには、サービス定義にserviceConnectConfigurationを、タスク定義にportMappingsを定義します。内部的には、verify.goのverifyServiceConnectConfigurationメソッドが設定を検証します。

flowchart TD TaskDef[タスク定義] --> |portMappings| Verify[検証] Service[サービス定義] --> |serviceConnectConfiguration| Verify Verify --> Deploy[デプロイ]

EBSボリュームのサポート

ecspressoはAmazon EBSボリュームをサポートしています。この機能はdeploy.goのApp.DeployメソッドとService構造体で実装されています。

設定するには、サービス定義にvolumeConfigurationsを、タスク定義にmountPointsとvolumesを定義します。内部的には、verify.goのverifyVolumeConfigurationsメソッドが設定を検証します。

タスク定義にportMappingsを、サービス定義にvpcLatticeConfigurationsを定義します。内部的には、verify.goのverifyVPCLatticeConfigurationsメソッドが設定を検証します。
