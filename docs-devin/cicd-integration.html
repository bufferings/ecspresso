4.1 CI/CDパイプラインとの統合

目次
  • 概要
  • GitHub Actionsとの統合
  • GitLab CIとの統合
  • CircleCIとの統合
  • ベストプラクティス

概要

ecspressoをCI/CDパイプラインに統合することで、ECSサービスのデプロイを自動化し、継続的デリバリーを実現できます。このページでは、一般的なCI/CDツールとecspressoを統合する方法を解説します。

CI/CDパイプラインでecspressoを使用する主なステップは以下の通りです：

  1. ecspressoのインストール
  2. AWS認証情報の設定
  3. 設定ファイルの検証（verify）
  4. 変更の差分確認（diff）
  5. デプロイの実行（deploy）
  6. デプロイ結果の確認（status）

GitHub Actionsとの統合

GitHub Actionsを使用してecspressoをCI/CDパイプラインに統合する方法を紹介します。

ワークフロー例

name: Deploy to ECS

on:
  push:
    branches: [ main ]
  workflow_dispatch:

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Set up Go
        uses: actions/setup-go@v4
        with:
          go-version: '1.20'
      
      - name: Install ecspresso
        run: |
          go install github.com/kayac/ecspresso/v2@latest
      
      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v2
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: ap-northeast-1
      
      - name: Verify resources
        run: ecspresso verify --config=./ecspresso.yml
      
      - name: Show diff
        run: ecspresso diff --config=./ecspresso.yml
      
      - name: Deploy to ECS
        run: ecspresso deploy --config=./ecspresso.yml

ポイント

  • GitHub Secretsを使用してAWS認証情報を安全に管理します。
  • デプロイ前にverifyとdiffコマンドを実行して、問題がないことを確認します。
  • workflow_dispatchを設定することで、手動でワークフローを実行することもできます。

GitLab CIとの統合

GitLab CIを使用してecspressoをCI/CDパイプラインに統合する方法を紹介します。

.gitlab-ci.yml例

stages:
  - verify
  - deploy

variables:
  GO_VERSION: "1.20"

before_script:
  - go install github.com/kayac/ecspresso/v2@latest

verify:
  stage: verify
  image: golang:${GO_VERSION}
  script:
    - ecspresso verify --config=./ecspresso.yml
    - ecspresso diff --config=./ecspresso.yml

deploy:
  stage: deploy
  image: golang:${GO_VERSION}
  script:
    - ecspresso deploy --config=./ecspresso.yml
  only:
    - main
  environment:
    name: production

ポイント

  • GitLab CIの環境変数を使用してAWS認証情報を設定します（AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY）。
  • ステージを分けることで、検証とデプロイを分離できます。
  • 環境を指定することで、デプロイ先を明確にします。

CircleCIとの統合

CircleCIを使用してecspressoをCI/CDパイプラインに統合する方法を紹介します。

config.yml例

version: 2.1

orbs:
  aws-cli: circleci/aws-cli@3.1

jobs:
  deploy:
    docker:
      - image: cimg/go:1.20
    steps:
      - checkout
      - aws-cli/setup:
          profile-name: default
      - run:
          name: Install ecspresso
          command: go install github.com/kayac/ecspresso/v2@latest
      - run:
          name: Verify resources
          command: ecspresso verify --config=./ecspresso.yml
      - run:
          name: Show diff
          command: ecspresso diff --config=./ecspresso.yml
      - run:
          name: Deploy to ECS
          command: ecspresso deploy --config=./ecspresso.yml

workflows:
  version: 2
  deploy-workflow:
    jobs:
      - deploy:
          filters:
            branches:
              only: main

ポイント

  • CircleCIのOrbsを使用してAWS CLIをセットアップします。
  • 環境変数を使用してAWS認証情報を設定します（AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY）。
  • ワークフローのフィルターを使用して、特定のブランチでのみデプロイを実行します。

ベストプラクティス

セキュリティ

  • 認証情報の保護: AWS認証情報はCI/CDツールのシークレット機能を使用して保護しましょう。
  • 最小権限の原則: デプロイに必要な最小限の権限のみを持つIAMユーザーまたはロールを使用しましょう。
  • 一時的な認証情報: 可能であれば、IAMロールと一時的な認証情報を使用しましょう。

パイプラインの設計

  • 段階的なデプロイ: 開発環境、ステージング環境、本番環境と段階的にデプロイするパイプラインを設計しましょう。
  • 自動テスト: デプロイ前に自動テストを実行して、アプリケーションの品質を確保しましょう。
  • 検証ステップ: verifyとdiffコマンドを必ず実行して、デプロイ前に問題を検出しましょう。

モニタリングと通知

  • デプロイ結果の通知: Slack、Teamsなどへの通知を設定して、デプロイ結果を関係者に共有しましょう。
  • デプロイ後の監視: デプロイ後にサービスの健全性を監視するステップを追加しましょう。
  • ロールバック自動化: 問題が検出された場合に自動的にロールバックするメカニズムを検討しましょう。

統合例：完全なワークフロー

以下は、ecspressoを使用した完全なCI/CDワークフローの例です：

graph TD A[コード変更] --> B[自動テスト] B --> C{テスト成功?} C -->|Yes| D[ecspresso verify] C -->|No| E[修正] E --> B D --> F{検証成功?} F -->|Yes| G[ecspresso diff] F -->|No| E G --> H[レビュー] H --> I{承認?} I -->|Yes| J[ecspresso deploy] I -->|No| E J --> K[デプロイ監視] K --> L{問題あり?} L -->|Yes| M[ecspresso rollback] L -->|No| N[完了] M --> N

このワークフローでは、コード変更からデプロイ、そして問題が発生した場合のロールバックまでの一連の流れを自動化しています。各ステップでの検証と承認プロセスにより、安全で信頼性の高いデプロイを実現します。
