コマンド概要

ecspressoは様々なコマンドを提供し、ECSサービスとタスクの管理を容易にします。各コマンドはcli.goで定義され、App構造体のメソッドとして実装されています。

flowchart TD CLI[CLI関数] --> Deploy[Deployコマンド] CLI --> Diff[Diffコマンド] CLI --> Verify[Verifyコマンド] CLI --> Run[Runコマンド] CLI --> Scale[Scaleコマンド] CLI --> Tasks[Tasksコマンド] CLI --> Exec[Execコマンド] CLI --> Other[その他のコマンド] Deploy --> DeployECS[ECSローリングデプロイ] Deploy --> DeployCodeDeploy[CodeDeployブルー/グリーン]

Deployコマンド (deploy.go)

deployコマンドはdeploy.goで実装されており、コンフィグレーションに基づいてECSサービスをデプロイします。

デプロイプロセスの内部動作：

  1. タスク定義ファイルからタスク定義を登録（--skip-task-definitionが指定されていない場合）
  2. サービス定義を更新（--no-update-serviceが指定されていない場合）
  3. サービスが安定状態になるまで待機

ブルー/グリーンデプロイメントの場合（deploymentController.typeがCODE_DEPLOYの場合）、以下のステップが実行されます：

  1. タスク定義を登録
  2. AppSpecファイルを生成
  3. CodeDeployアプリケーションとデプロイメントグループが存在することを確認
  4. CodeDeployデプロイメントを作成
  5. デプロイメントが完了するまで待機

ブルー/グリーンデプロイメント（AWS CodeDeployを使用）の場合：

ecspresso deployはCODE_DEPLOYデプロイメントコントローラーを使用してサービスをデプロイできます。ecs-service-def.jsonを以下のように設定します：

{
  "deploymentController": {
    "type": "CODE_DEPLOY"
  },
  // ...
}

主なオプション：

  • --wait-until: デプロイメントを待機する条件を指定（stableまたはdeployed）
  • --revision: タスク定義の特定のリビジョンを指定
  • --latest-task-definition: タスク定義の最新リビジョンを使用
  • --skip-task-definition: 新しいタスク定義の登録をスキップ
  • --force-new-deployment: サービス定義とタスク定義が変更されていなくても、新しいデプロイメントを強制

Diffコマンド (diff.go)

diffコマンドはdiff.goで実装されており、ローカルの定義と実際にデプロイされているリソースを比較します：

  • タスク定義の差分
  • サービス定義の差分
  • IAMロールなどの関連リソースの差分
sequenceDiagram participant User as ユーザー participant App as ecspresso participant AWS as AWS API User->>App: ecspresso diff App->>AWS: DescribeService AWS-->>App: サービス情報 App->>AWS: DescribeTaskDefinition AWS-->>App: タスク定義情報 App->>App: ローカル定義読込 App->>App: 定義を正規化 App->>App: 差分計算 App-->>User: 差分表示

差分表示はGitスタイルの統一差分形式で表示され、差分結果はServiceForDiff構造体とNormalizeServiceメソッドを使用して比較前に正規化されます。内部的にはApp.Diffメソッドが以下のステップを実行します：

  1. 現在のサービスとタスク定義を取得
  2. ローカルの定義を読み込み
  3. 比較のために両方を正規化
  4. 差分を計算して表示

Verifyコマンド (verify.go)

verifyコマンドはverify.goで実装されており、サービス定義とタスク定義に関連するリソースが正しく構成されていることを検証します。内部的にはApp.Verifyメソッドがverifier構造体を使用して検証を実行します。

flowchart TD Verify[Verify] --> Cluster[クラスター検証] Verify --> TargetGroups[ターゲットグループ検証] Verify --> IAMRoles[IAMロール検証] Verify --> Images[コンテナイメージ検証] Verify --> Secrets[シークレット検証] Verify --> Logs[ログ設定検証]

検証プロセスは以下のステップで行われます：

  1. 検証状態の初期化（verifyState構造体）
  2. 各リソースタイプごとの検証関数の実行
  3. 検証結果のキャッシュと再利用（パフォーマンス向上のため）

例えば、以下のリソースを検証します：

  • ECSクラスターが存在するか（verifyClusterメソッド）
  • サービス定義内のターゲットグループが、定義内のコンテナ名とポートと一致するか（verifyTargetGroupsメソッド）
  • タスクロールとタスク実行ロールが存在し、ecs-tasks.amazonaws.comによって引き受けられるか（verifyIAMRoleメソッド）
  • タスク定義で定義されたURLにコンテナイメージが存在するか（verifyContainerImageメソッド）
  • タスク定義内のシークレットが存在し、読み取り可能か（verifySecretsメソッド）
  • 指定されたCloudWatchロググループストリームにログストリームを作成し、メッセージを送信できるか（verifyLogConfigurationメソッド）

Runコマンド (run.go)

runコマンドはrun.goで実装されており、指定されたタスク定義を使用して一時的なタスクを実行します。内部的にはApp.Runメソッドが以下のステップを実行します：

sequenceDiagram participant User as ユーザー participant App as ecspresso participant AWS as AWS API User->>App: ecspresso run App->>App: タスク定義解決 opt タスク定義登録が必要 App->>AWS: RegisterTaskDefinition AWS-->>App: タスク定義ARN end App->>App: 起動タイプ決定 App->>AWS: RunTask AWS-->>App: タスク情報 opt 待機が必要 App->>AWS: DescribeTasks（繰り返し） AWS-->>App: タスク状態 end App-->>User: 完了メッセージ
  1. タスク定義の解決（--task-defが指定されていない場合はサービスのタスク定義を使用）
  2. タスク定義の登録（--register-task-definitionが指定されている場合）
  3. 起動タイプとネットワーク設定の決定（EC2、Fargate、または外部）
  4. RunTaskリクエストの構築と送信
  5. タスクの起動と状態の監視（--waitが指定されている場合）

コマンド例：

$ ecspresso run --config ecspresso.yml --task-def=db-migrate.json --overrides='{"containerOverrides":[{"name":"app","command":["bundle","exec","rake","db:migrate"]}]}'

--task-defが設定されていない場合、ecspressoはサービスに含まれるタスク定義を使用します。--overridesオプションを使用すると、コンテナの環境変数やコマンドなどを上書きできます。

Scaleコマンド (deploy.go)

サービスの希望するカウント数を変更します。内部的にはdeploy.goのApp.Deployメソッドを特定のオプションで呼び出します。

$ ecspresso scale --tasks 10 --auto-scaling-min 5 --auto-scaling-max 20

scaleコマンドはdeploy --skip-task-definition --no-update-serviceと同等です。--auto-scaling-minと--auto-scaling-maxオプションを使用すると、Application Auto Scalingの設定も更新できます。

Tasksコマンド (tasks.go)

tasksコマンドはtasks.goで実装されており、サービスによって実行されるタスク、またはタスク定義と同じファミリーを持つタスクを一覧表示します。内部的にはApp.Tasksメソッドが以下のステップを実行します：

  1. サービスまたはタスク定義ファミリーに関連するタスクを検索
  2. タスクの詳細情報（ステータス、起動時間、IPアドレスなど）を取得
  3. フィルターコマンド（ECSPRESSO_FILTER_COMMAND環境変数で指定）を使用してタスクを選択
  4. 選択されたタスクに対して操作（停止、情報表示など）を実行

コマンド例：

$ ECSPRESSO_FILTER_COMMAND=peco ecspresso tasks --find --stop

--findオプションを使用すると、フィルターコマンドを使用してタスクを選択できます。--stopオプションを追加すると、選択したタスクを停止できます。

Execコマンド (exec.go)

execコマンドはexec.goで実装されており、実行中のタスクにSSMセッションマネージャーを通じて接続し、コマンドを実行できます。内部的にはApp.Execメソッドが以下のステップを実行します：

flowchart TD Exec[Exec] --> Select[タスク選択] Select --> Session[SSMセッション構築] Session --> Connect[接続確立] Connect --> Command[コマンド実行] Connect --> Shell[インタラクティブシェル] Connect --> Forward[ポートフォワーディング]
  1. タスクの選択（--taskが指定されていない場合、フィルターコマンドを使用してタスクを選択）
  2. SSMセッションドキュメントの構築
  3. SSMセッションの開始
  4. 必要に応じてポートフォワーディングの設定
  5. コマンド実行またはインタラクティブシェルの提供

コマンド例：

$ ecspresso exec --port-forward --port 80 --local-port 8080 --container app

--port-forwardオプションを使用すると、コンテナのポートをローカルポートにフォワードできます。--containerオプションを使用すると、複数コンテナのタスクで特定のコンテナを指定できます。

各コマンドの詳細については、ecspresso command --helpを参照してください。内部実装の詳細はGitHubリポジトリのソースコードを参照してください。
