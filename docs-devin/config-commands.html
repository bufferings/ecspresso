3.3 設定・検証コマンド

目次

  • diffコマンド
  • renderコマンド
  • statusコマンド

initコマンド

ecspressoの設定ファイルを初期化します。既存のECSサービスからecspresso用の設定ファイルを生成するために使用します。

基本的な使い方

$ ecspresso init [オプション]

主なオプション

  • --region REGION: AWSリージョンを指定します。
  • --cluster CLUSTER: ECSクラスター名を指定します。
  • --service SERVICE: ECSサービス名を指定します。
  • --config FILE: 設定ファイル名を指定します（デフォルト: ecspresso.yml）。
  • --task-definition ARN: 特定のタスク定義ARNを指定します。
  • --no-service: サービスなしでタスク定義のみを初期化します。

使用例

$ ecspresso init --region ap-northeast-1 --cluster my-cluster --service my-service

指定したクラスターとサービスに基づいて設定ファイルを初期化します。これにより、以下のファイルが生成されます：

  • ecspresso.yml: ecspressoの設定ファイル
  • ecs-service-def.json: ECSサービス定義
  • ecs-task-def.json: ECSタスク定義

verifyコマンド

デプロイ前にリソースの設定が正しいか検証します。タスク定義やサービス定義で参照されているリソース（IAMロール、ログ設定、シークレットなど）が実際に存在するかを確認します。

基本的な使い方

$ ecspresso verify [オプション]

主なオプション

  • --task-definition FILE: 検証するタスク定義ファイルを指定します。
  • --no-cache: 検証キャッシュを使用しません。
  • --secrets: シークレットの存在も検証します。
  • --only-task-definition: タスク定義のみを検証します。
  • --only-service-definition: サービス定義のみを検証します。

検証項目

  • IAMロール（タスク実行ロール、タスクロール）
  • CloudWatch Logsのロググループ
  • Secrets ManagerやSSMのシークレット（--secretsオプション使用時）
  • ロードバランサーのターゲットグループ
  • セキュリティグループ
  • サブネット

使用例

$ ecspresso verify --secrets

シークレットの存在も含めてリソースを検証します。問題がなければ成功メッセージが表示され、問題があれば詳細なエラーメッセージが表示されます。

diffコマンド

ローカルの設定とAWS上の設定の差分を表示します。タスク定義やサービス定義の変更点を確認するのに役立ちます。

基本的な使い方

$ ecspresso diff [オプション]

主なオプション

  • --no-unified: 統一差分形式を使用しません。
  • --external COMMAND: 差分表示に外部コマンドを使用します。
  • --only-task-definition: タスク定義の差分のみを表示します。
  • --only-service-definition: サービス定義の差分のみを表示します。

使用例

$ ecspresso diff --external "colordiff -u"

colordiffを使用して差分を色付きで表示します。

$ ecspresso diff --only-task-definition

タスク定義の差分のみを表示します。

renderコマンド

タスク定義やサービス定義をレンダリングして表示します。テンプレート関数やプラグインが適用された後の最終的な定義を確認できます。

基本的な使い方

$ ecspresso render [オプション]

主なオプション

  • --task-definition: タスク定義をレンダリングします。
  • --service-definition: サービス定義をレンダリングします。
  • --appspec: CodeDeployのAppSpecをレンダリングします。

使用例

$ ecspresso render --task-definition > rendered-task-def.json

タスク定義をレンダリングしてファイルに保存します。

$ ecspresso render --service-definition | jq

サービス定義をレンダリングしてjqで整形表示します。

statusコマンド

ECSサービスの現在の状態を表示します。デプロイ状況の確認やトラブルシューティングに役立ちます。

基本的な使い方

$ ecspresso status [オプション]

主なオプション

  • --events N: 表示するイベント数を指定します（デフォルト: 10）。
  • --tasks: タスクの状態も表示します。
  • --output FORMAT: 出力形式を指定します（json, yaml, table）。

表示される情報

  • サービス名、クラスター名
  • タスク定義ARN
  • 実行中のタスク数、希望するタスク数
  • デプロイメント状態（PRIMARY, ACTIVE）
  • 最近のサービスイベント
  • タスクの状態（--tasksオプション使用時）

使用例

$ ecspresso status --events 10 --tasks

サービスの状態と直近10件のイベント、およびタスクの状態を表示します。

ベストプラクティス

  • デプロイ前の検証: デプロイ前に必ずverifyコマンドを実行して、リソースの設定が正しいことを確認しましょう。
  • 差分確認: デプロイ前にdiffコマンドで変更内容を確認しましょう。
  • CI/CDパイプラインでの活用: verifyとdiffコマンドをCI/CDパイプラインに組み込むことで、問題のある変更を早期に検出できます。
  • レンダリング結果の確認: 複雑なテンプレート関数やプラグインを使用している場合は、renderコマンドで最終的な定義を確認しましょう。
