deployコマンド

deployコマンドは、ECSサービスをデプロイするために使用します。タスク定義の登録とサービスの更新を行います。

基本的な使い方

ecspresso deploy [オプション]

主なオプション

オプション 説明 デフォルト値
--dry-run 実際にデプロイせずに、実行されるアクションを表示する false
--force-new-deployment 新しいデプロイメントを強制する false
--no-wait-until-stable サービスが安定するまで待機しない false
--wait-until 待機条件（deployed または stable） stable
--suspend-auto-scaling デプロイ中にApplication Auto Scalingを一時停止する false
--update-service サービスの更新のみを行う（タスク定義は更新しない） false
--skip-task-definition タスク定義の登録をスキップする false
--revision 使用するタスク定義のリビジョン -
--latest-task-definition 最新のタスク定義を使用する false
--rollback-events 自動ロールバックをトリガーするイベント -
--tasks デプロイするタスク数 -

処理フロー

deployコマンドの基本的な処理フロー：

flowchart TD A[Deploy Command] --> B[Load Config] B --> C[Register Task] C --> D[Update Service] D --> E[Wait for Stability]

使用例

# 基本的なデプロイ
ecspresso deploy

# ドライランモード（実際にデプロイしない）
ecspresso deploy --dry-run

# 新しいデプロイメントを強制する
ecspresso deploy --force-new-deployment

# サービスが安定するまで待機しない
ecspresso deploy --no-wait-until-stable

# デプロイ中にApplication Auto Scalingを一時停止する
ecspresso deploy --suspend-auto-scaling

# 特定のリビジョンのタスク定義を使用する
ecspresso deploy --revision 10

# 最新のタスク定義を使用する
ecspresso deploy --latest-task-definition

# タスク数を指定してデプロイする
ecspresso deploy --tasks 5

rollbackコマンド

rollbackコマンドは、ECSサービスを以前のバージョンにロールバックするために使用します。このコマンドはrollback.goファイルに実装されています。

基本的な使い方

ecspresso rollback [オプション]

主なオプション

オプション 説明 デフォルト値
--dry-run 実際にロールバックせずに、実行されるアクションを表示する false
--deregister ロールバック後に現在のタスク定義を登録解除する false
--no-wait-until-stable サービスが安定するまで待機しない false
--revision ロールバックするタスク定義のリビジョン -

ロールバックプロセス

rollbackコマンドの内部処理フロー：

flowchart TD A[ロールバック開始] --> B[設定読み込み] B --> C[現在のサービス取得] C --> D[以前のタスク定義取得] D --> E[サービス更新] E --> F[安定性確認]

使用例

# 基本的なロールバック（前回のタスク定義に戻す）
ecspresso rollback

# ドライランモード（実際にロールバックしない）
ecspresso rollback --dry-run

# 特定のリビジョンにロールバックする
ecspresso rollback --revision 15

# ロールバック後に現在のタスク定義を登録解除する
ecspresso rollback --deregister

# サービスが安定するまで待機しない
ecspresso rollback --no-wait-until-stable

scaleコマンド

scaleコマンドは、ECSサービスのタスク数を変更するために使用します。このコマンドはdeploy.goファイルに実装されています。

基本的な使い方

ecspresso scale --tasks N [オプション]

主なオプション

オプション 説明 デフォルト値
--tasks 設定するタスク数（必須） -
--dry-run 実際にスケールせずに、実行されるアクションを表示する false
--no-wait-until-stable サービスが安定するまで待機しない false

スケーリングプロセス

scaleコマンドの内部処理フロー：

flowchart TD A[スケール開始] --> B[設定読み込み] B --> C[現在のサービス取得] C --> D[タスク数更新] D --> E[安定性確認]

使用例

# タスク数を5に設定
ecspresso scale --tasks 5

# ドライランモード（実際にスケールしない）
ecspresso scale --tasks 10 --dry-run

# タスク数を0に設定（サービスを停止）
ecspresso scale --tasks 0

# サービスが安定するまで待機しない
ecspresso scale --tasks 3 --no-wait-until-stable

waitコマンド

waitコマンドは、ECSサービスが安定するまで待機するために使用します。このコマンドはwait.goファイルに実装されています。

基本的な使い方

ecspresso wait [オプション]

主なオプション

オプション 説明 デフォルト値
--timeout タイムアウト時間（秒） 600
--wait-until 待機条件（deployed または stable） stable

待機プロセス

waitコマンドの内部処理フロー：

flowchart TD A[Wait開始] --> B[サービス状態取得] B --> C{安定している?} C -->|はい| D[完了] C -->|いいえ| E[待機] E --> F{タイムアウト?} F -->|はい| G[エラー] F -->|いいえ| B

使用例

# サービスが安定するまで待機
ecspresso wait

# デプロイメントが完了するまで待機
ecspresso wait --wait-until deployed

# タイムアウト時間を300秒に設定
ecspresso wait --timeout 300

デプロイ戦略

ecspressoは以下の2つのデプロイ戦略をサポートしています：

flowchart TD A[デプロイ戦略] --> B[ECSローリングデプロイ] A --> C[CodeDeployブルー/グリーン] B --> D[タスクを徐々に置き換え] C --> E[新環境を作成し徐々に移行] D --> F[maximumPercent] D --> G[minimumHealthyPercent] D --> H[deploymentCircuitBreaker] E --> I[トラフィック分割] E --> J[デプロイフック] E --> K[ロールバック容易]

ECSローリングデプロイ

ECSローリングデプロイは、タスクを徐々に置き換えていく戦略です。以下のパラメータで設定できます：

# ecspresso.yml
service_definition: ecs-service-def.json

# ecs-service-def.json
{
  "deploymentConfiguration": {
    "maximumPercent": 200,
    "minimumHealthyPercent": 100,
    "deploymentCircuitBreaker": {
      "enable": true,
      "rollback": true
    }
  }
}

CodeDeployブルー/グリーンデプロイ

CodeDeployブルー/グリーンデプロイは、新しい環境（グリーン）を作成し、トラフィックを徐々に移行する戦略です。以下のパラメータで設定できます：

# ecspresso.yml
service_definition: ecs-service-def.json
appspec: appspec.yaml

# ecs-service-def.json
{
  "deploymentController": {
    "type": "CODE_DEPLOY"
  }
}

# appspec.yaml
version: 0.0
Resources:
  - TargetService:
      Type: AWS::ECS::Service
      Properties:
        TaskDefinition: "<TASK_DEFINITION>"
        LoadBalancerInfo:
          ContainerName: "web"
          ContainerPort: 80
        PlatformVersion: "1.4.0"
Hooks:
  - BeforeInstall: "LambdaFunctionToValidateBeforeInstall"
  - AfterInstall: "LambdaFunctionToValidateAfterInstall"
  - AfterAllowTestTraffic: "LambdaFunctionToValidateAfterTestTrafficStarts"
  - BeforeAllowTraffic: "LambdaFunctionToValidateBeforeAllowingProductionTraffic"
  - AfterAllowTraffic: "LambdaFunctionToValidateAfterAllowingProductionTraffic"

ベストプラクティス

デプロイメントコマンドを使用する際のベストプラクティス：

  • デプロイ前の検証: デプロイ前にecspresso verifyとecspresso diffを実行して、設定が正しいことを確認します。
  • ドライランの活用: 本番環境でデプロイする前に--dry-runオプションを使用して、実行されるアクションを確認します。
  • ロールバック計画: デプロイに問題が発生した場合に備えて、ロールバック手順を準備しておきます。
  • デプロイサーキットブレーカーの使用: ECSローリングデプロイでは、デプロイサーキットブレーカーを有効にして、問題が発生した場合に自動的にロールバックするようにします。
  • 段階的なデプロイ: 重要な変更を行う場合は、まず一部のタスクにデプロイし、問題がないことを確認してから全体にデプロイします。
  • モニタリングの強化: デプロイ中は、CloudWatchやX-Rayなどのモニタリングツールを使用して、アプリケーションの状態を監視します。
  • CI/CDパイプラインの統合: デプロイプロセスをCI/CDパイプラインに統合して、自動化します。
  • 環境変数の活用: 環境ごとに異なる設定を環境変数で管理し、テンプレート関数で参照します。
