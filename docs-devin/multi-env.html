概要

多くの組織では、開発環境、ステージング環境、本番環境など、複数の環境でECSサービスを運用しています。ecspressoは、これらの環境間で一貫したデプロイを実現するための機能を提供します。このページでは、ecspressoを使用して複数環境を効率的に管理する方法を解説します。

複数環境を管理する際の主な課題は以下の通りです：

  • 環境ごとに異なる設定値の管理
  • 環境間での設定の一貫性の確保
  • 環境固有のリソースへの参照
  • 環境ごとのデプロイプロセスの管理

環境管理の方法

ecspressoでは、以下の方法で複数環境を管理できます：

1. 環境ごとの設定ファイル

環境ごとに別々の設定ファイルを用意する方法です。


# 開発環境用
$ ecspresso deploy --config ecspresso-dev.yml

# ステージング環境用
$ ecspresso deploy --config ecspresso-staging.yml

# 本番環境用
$ ecspresso deploy --config ecspresso-prod.yml

2. 環境変数ファイル

共通の設定ファイルと環境ごとの環境変数ファイルを組み合わせる方法です。


# 開発環境用
$ ecspresso deploy --config ecspresso.yml --env-file dev.env

# ステージング環境用
$ ecspresso deploy --config ecspresso.yml --env-file staging.env

# 本番環境用
$ ecspresso deploy --config ecspresso.yml --env-file prod.env

3. Jsonnetによる環境分離

Jsonnetを使用して、環境ごとの設定を一元管理する方法です。


# 開発環境用
$ ecspresso deploy --config ecspresso.jsonnet --env ENV=dev

# ステージング環境用
$ ecspresso deploy --config ecspresso.jsonnet --env ENV=staging

# 本番環境用
$ ecspresso deploy --config ecspresso.jsonnet --env ENV=prod

環境変数の活用

ecspressoの設定ファイルでは、テンプレート関数を使用して環境変数を参照できます。これにより、環境固有の値を外部化できます。

テンプレート関数の例

{
  "cluster": "{{ env `CLUSTER_NAME` }}",
  "service": "{{ env `SERVICE_NAME` }}",
  "service_definition": "{{ env `SERVICE_DEF_PATH` }}",
  "task_definition": "{{ env `TASK_DEF_PATH` }}",
  "timeout": "{{ env `DEPLOY_TIMEOUT` `10m` }}"
}

環境変数ファイルの例

dev.env:

CLUSTER_NAME=my-dev-cluster
SERVICE_NAME=my-service-dev
SERVICE_DEF_PATH=./service-def-dev.json
TASK_DEF_PATH=./task-def-dev.json
DEPLOY_TIMEOUT=5m

prod.env:

CLUSTER_NAME=my-prod-cluster
SERVICE_NAME=my-service-prod
SERVICE_DEF_PATH=./service-def-prod.json
TASK_DEF_PATH=./task-def-prod.json
DEPLOY_TIMEOUT=15m

Jsonnetによる環境設定

Jsonnetは、JSONを拡張したテンプレート言語で、環境ごとの設定を効率的に管理できます。

Jsonnetの例

local env = std.extVar('ENV');

local envConfig = {
  dev: {
    cluster: 'my-dev-cluster',
    service: 'my-service-dev',
    desiredCount: 1,
    memory: '512',
    cpu: '256',
  },
  staging: {
    cluster: 'my-staging-cluster',
    service: 'my-service-staging',
    desiredCount: 2,
    memory: '1024',
    cpu: '512',
  },
  prod: {
    cluster: 'my-prod-cluster',
    service: 'my-service-prod',
    desiredCount: 3,
    memory: '2048',
    cpu: '1024',
  },
};

local config = envConfig[env];

{
  cluster: config.cluster,
  service: config.service,
  service_definition: 'ecs-service-def.json',
  task_definition: 'ecs-task-def.json',
  timeout: '10m',
}

タスク定義でのJsonnet活用例

local env = std.extVar('ENV');

local envConfig = {
  dev: {
    memory: '512',
    cpu: '256',
    logGroup: '/ecs/my-service-dev',
  },
  staging: {
    memory: '1024',
    cpu: '512',
    logGroup: '/ecs/my-service-staging',
  },
  prod: {
    memory: '2048',
    cpu: '1024',
    logGroup: '/ecs/my-service-prod',
  },
};

local config = envConfig[env];

{
  family: 'my-service-' + env,
  containerDefinitions: [
    {
      name: 'app',
      image: 'my-image:latest',
      memory: config.memory,
      cpu: config.cpu,
      logConfiguration: {
        logDriver: 'awslogs',
        options: {
          'awslogs-group': config.logGroup,
          'awslogs-region': 'ap-northeast-1',
          'awslogs-stream-prefix': 'ecs',
        },
      },
    },
  ],
}

プラグインの活用

ecspressoのプラグインを活用することで、環境固有のリソースを参照できます。

SSMパラメータストアの活用

{
  "plugins": [
    {
      "name": "ssm",
      "config": {
        "region": "ap-northeast-1"
      }
    }
  ]
}

タスク定義での使用例：

{
  "containerDefinitions": [
    {
      "name": "app",
      "image": "{{ ssm `/my-service/${ENV}/image` }}",
      "environment": [
        {
          "name": "DB_HOST",
          "value": "{{ ssm `/my-service/${ENV}/db-host` }}"
        },
        {
          "name": "API_KEY",
          "value": "{{ ssm `/my-service/${ENV}/api-key` }}"
        }
      ]
    }
  ]
}

Secrets Managerの活用

{
  "plugins": [
    {
      "name": "secrets",
      "config": {
        "region": "ap-northeast-1"
      }
    }
  ]
}

タスク定義での使用例：

{
  "containerDefinitions": [
    {
      "name": "app",
      "secrets": [
        {
          "name": "DB_PASSWORD",
          "valueFrom": "{{ secrets `my-service/${ENV}/db-password` }}"
        },
        {
          "name": "API_SECRET",
          "valueFrom": "{{ secrets `my-service/${ENV}/api-secret` }}"
        }
      ]
    }
  ]
}

ベストプラクティス

環境設定の管理

  • DRYの原則: 環境間で共通の設定は一箇所で管理し、環境固有の設定のみを分離しましょう。
  • 設定の検証: 各環境の設定をverifyコマンドで検証して、問題を早期に発見しましょう。
  • 環境変数の命名規則: 環境変数には明確な命名規則を設け、環境ごとの違いを分かりやすくしましょう。

デプロイプロセス

  • 段階的なデプロイ: 開発環境→ステージング環境→本番環境と段階的にデプロイし、各段階で検証しましょう。
  • 環境ごとの承認プロセス: 本番環境へのデプロイには、追加の承認ステップを設けましょう。
  • ロールバック計画: 各環境でのロールバック手順を明確にしておきましょう。

セキュリティ

  • 環境分離: 環境ごとにAWSアカウントを分離し、本番環境へのアクセスを制限しましょう。
  • シークレット管理: 環境ごとにシークレットを分離し、Secrets ManagerやSSMパラメータストアで管理しましょう。
  • 最小権限の原則: 各環境のIAMロールには、必要最小限の権限のみを付与しましょう。

実装例：完全な環境分離

以下は、開発、ステージング、本番の3つの環境を完全に分離した実装例です：

ディレクトリ構造

my-service/
├── common/
│   ├── task-def-base.jsonnet
│   └── service-def-base.jsonnet
├── dev/
│   ├── ecspresso.yml
│   ├── env.sh
│   ├── task-def.jsonnet
│   └── service-def.jsonnet
├── staging/
│   ├── ecspresso.yml
│   ├── env.sh
│   ├── task-def.jsonnet
│   └── service-def.jsonnet
└── prod/
    ├── ecspresso.yml
    ├── env.sh
    ├── task-def.jsonnet
    └── service-def.jsonnet

デプロイスクリプト例

#!/bin/bash

ENV=$1
if [ -z "$ENV" ]; then
  echo "Usage: $0 "
  echo "  env: dev, staging, prod"
  exit 1
fi

cd $ENV
source env.sh

echo "Deploying to $ENV environment..."
ecspresso verify
ecspresso diff
ecspresso deploy

このような構造により、環境ごとの設定を明確に分離しつつ、共通の設定を再利用できます。
