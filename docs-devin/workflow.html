一般的なワークフロー

ecspressoを使用した一般的なワークフローは以下のようになります：

flowchart TD Init[初期設定] --> Config[設定ファイル作成] Config --> TaskDef[タスク定義作成] TaskDef --> ServiceDef[サービス定義作成] ServiceDef --> Verify[リソース検証] Verify --> Deploy[デプロイ] Deploy --> Monitor[モニタリング] Monitor --> Update[更新] Update --> Verify
  1. 初期設定: ecspresso initコマンドを使用して、プロジェクトの初期設定を行います。
  2. 設定ファイル作成: ecspresso.ymlファイルを作成し、クラスター名やサービス名などの基本設定を行います。
  3. タスク定義作成: JSONまたはYAMLでタスク定義ファイルを作成します。
  4. サービス定義作成: JSONまたはYAMLでサービス定義ファイルを作成します。
  5. リソース検証: ecspresso verifyコマンドを使用して、設定が正しいことを確認します。
  6. デプロイ: ecspresso deployコマンドを使用して、サービスをデプロイします。
  7. モニタリング: ecspresso statusコマンドを使用して、サービスの状態を確認します。
  8. 更新: 必要に応じてタスク定義やサービス定義を更新し、再デプロイします。

CI/CD統合

ecspressoはCI/CDパイプラインに簡単に統合できます。以下は一般的な統合パターンです：

sequenceDiagram participant Dev as 開発者 participant CI as CI/CDシステム participant ECR as Amazon ECR participant ECS as Amazon ECS Dev->>CI: コード変更をプッシュ CI->>CI: ビルドとテスト CI->>ECR: Dockerイメージをプッシュ CI->>CI: ecspressoで設定を更新 CI->>CI: ecspresso verify CI->>ECS: ecspresso deploy ECS-->>CI: デプロイ結果 CI-->>Dev: デプロイ通知

CI/CDパイプラインでecspressoを使用する際の主なステップ：

  1. CI/CD環境にecspressoをインストールします。
  2. AWS認証情報を設定します（IAMロールまたは環境変数）。
  3. 必要に応じてタスク定義のイメージタグを更新します。
  4. ecspresso verifyを実行して設定を検証します。
  5. ecspresso deployを実行してサービスをデプロイします。
  6. オプションでecspresso waitを実行してデプロイの完了を待ちます。

GitHub Actionsでの使用例

GitHub Actionsでecspressoを使用する例：

name: Deploy to ECS

on:
  push:
    branches: [ main ]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v1
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: ap-northeast-1
      
      - name: Login to Amazon ECR
        id: login-ecr
        uses: aws-actions/amazon-ecr-login@v1
      
      - name: Build and push Docker image
        env:
          ECR_REGISTRY: ${{ steps.login-ecr.outputs.registry }}
          ECR_REPOSITORY: my-app
          IMAGE_TAG: ${{ github.sha }}
        run: |
          docker build -t $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG .
          docker push $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG
      
      - name: Install ecspresso
        run: |
          curl -sL https://github.com/kayac/ecspresso/releases/download/v2.0.0/ecspresso_2.0.0_linux_amd64.tar.gz | tar xzf -
          sudo mv ecspresso /usr/local/bin/
      
      - name: Update task definition
        env:
          ECR_REGISTRY: ${{ steps.login-ecr.outputs.registry }}
          ECR_REPOSITORY: my-app
          IMAGE_TAG: ${{ github.sha }}
        run: |
          sed -i "s|{{IMAGE_URI}}|$ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG|g" ecs-task-def.json
      
      - name: Deploy to ECS
        run: |
          ecspresso verify
          ecspresso deploy --no-wait-until-stable
          ecspresso wait

GitLab CIでの使用例

GitLab CIでecspressoを使用する例：

stages:
  - build
  - deploy

variables:
  AWS_DEFAULT_REGION: ap-northeast-1
  ECR_REPOSITORY: my-app

build:
  stage: build
  image: docker:latest
  services:
    - docker:dind
  script:
    - apk add --no-cache aws-cli
    - aws ecr get-login-password | docker login --username AWS --password-stdin $ECR_REGISTRY
    - docker build -t $ECR_REGISTRY/$ECR_REPOSITORY:$CI_COMMIT_SHA .
    - docker push $ECR_REGISTRY/$ECR_REPOSITORY:$CI_COMMIT_SHA
  only:
    - main

deploy:
  stage: deploy
  image: alpine:latest
  script:
    - apk add --no-cache curl aws-cli jq
    - curl -sL https://github.com/kayac/ecspresso/releases/download/v2.0.0/ecspresso_2.0.0_linux_amd64.tar.gz | tar xzf -
    - mv ecspresso /usr/local/bin/
    - sed -i "s|{{IMAGE_URI}}|$ECR_REGISTRY/$ECR_REPOSITORY:$CI_COMMIT_SHA|g" ecs-task-def.json
    - ecspresso verify
    - ecspresso deploy
  only:
    - main

ベストプラクティス

ecspressoをCI/CDパイプラインで使用する際のベストプラクティス：

  • 環境変数の活用: 環境ごとに異なる設定を環境変数で管理し、テンプレート関数で参照します。
  • 検証の徹底: デプロイ前に必ずecspresso verifyを実行して設定を検証します。
  • 差分確認: ecspresso diffを使用して、変更内容を事前に確認します。
  • ロールバック計画: 問題が発生した場合にecspresso rollbackを使用できるようにしておきます。
  • 待機戦略: --no-wait-until-stableとecspresso waitを組み合わせて、長時間のデプロイでもCIがタイムアウトしないようにします。
  • ログの保存: デプロイログを保存して、問題が発生した場合に調査できるようにします。
  • 通知の設定: デプロイの成功/失敗をSlackなどに通知するようにします。
